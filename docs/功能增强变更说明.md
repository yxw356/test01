## 2. 对话落库与检索引用

### 2.1 设计说明

- **Redis**：仍用于多轮上下文（近 7 天、最多 20 条消息），保证追问体验。
- **MySQL `conversations` 表**：每轮问答持久化，供历史查询与合规留存。
- **结构化引用**：RAG 最终 Top-K 文档块序列化为 JSON，与问答一并存储。

### 2.2 数据表变更

`conversations` 表新增字段（JPA `ddl-auto: update` 或执行 `docs/databases/ddl.sql`）：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| session_id | VARCHAR(64) | Redis 会话 ID，关联多轮上下文 |
| retrieval_citations | TEXT | 检索引用 JSON 数组 |

引用 JSON 单条结构：

```json
{
  "index": 1,
  "fileMd5": "abc123...",
  "fileName": "报销制度.pdf",
  "chunkId": 3,
  "parentId": "parent-001",
  "score": 0.92,
  "snippet": "审批流程需经部门负责人..."
}
```

## 3. 知识库索引状态

### 3.1 状态定义

`file_upload.index_status`（与上传状态 `status` 分离）：

| 值 | 常量 | 含义 |
| --- | --- | --- |
| 0 | PENDING | 已合并，等待 Kafka 消费 |
| 1 | INDEXING | 正在解析 / 向量化 |
| 2 | INDEXED | 已写入 ES，可检索（历史数据默认 2） |
| 3 | FAILED | 索引失败，见 `index_error` |

### 3.2 状态流转

```text
上传合并成功 → PENDING（UploadController 发 Kafka 后）
     ↓
Kafka 消费开始 → INDEXING（FileProcessingConsumer）
     ↓
向量化成功 → INDEXED
     ↓（异常）
索引失败 → FAILED（写入 index_error）
```

### 3.3 后端要点

| 类 | 说明 |
| --- | --- |
| `FileIndexStatus` | 状态常量 |
| `FileIndexStatusService` | 更新 `index_status` / `index_error` |
| `FileUpload` | 新增 `indexStatus`、`indexError` |
| `FileProcessingConsumer` | 消费各阶段更新状态 |
| `DocumentController` | 列表 API 返回 `indexStatus`、`indexError` |

### 3.4 前端要点

| 文件 | 变更 |
| --- | --- |
| `enum/index.ts` | 新增 `IndexStatus` 枚举 |
| `knowledge-base/index.vue` | 「索引状态」列；概览「可检索 / 索引中」；待索引时每 5s 自动刷新 |

## 4. 索引失败重试

### 4.1 API

```http
POST /api/v1/documents/{fileMd5}/reindex
Authorization: Bearer <TOKEN>
```

- **权限**：文件所有者或 `ADMIN`
- **逻辑**：重新生成 MinIO 预签名 URL → 投递 Kafka → 状态置为 `PENDING`
- **前置**：上传状态须为已完成（`status = 1`）

### 4.2 后端要点

| 类 | 说明 |
| --- | --- |
| `DocumentIndexService` | 重试调度核心逻辑 |
| `DocumentController` | `POST /{fileMd5}/reindex` |
| `OrgTagAuthorizationFilter` | 放行 `POST .../reindex` |

### 4.3 前端

知识库列表：索引状态为 **待索引 / 索引中 / 索引失败** 时显示「重试索引」按钮。

## 6. Bug 修复：运行监控刷新 500

**现象**：运行监控页点击「刷新」返回 `Request failed with status code 500`。

**原因**：`MonitoringService.collectStatus()` 使用 `Map.of()` 组装 metrics，其中 `lastIndexFailureMessage`、`lastIndexFailureAt` 在未发生索引失败时为 `null`；`Map.of` 不允许 null 值，触发 `NullPointerException`。

**修复**：改用 `LinkedHashMap`，空值转为空字符串；`Instant` 格式化为字符串。

**涉及文件**：`MonitoringService.java`、`AdminOpsController.java`（补充错误日志）。

## 9. 验收步骤

### 9.1 索引状态

1. 登录 → 知识库 → 上传 txt/pdf  
2. 观察「索引状态」：待索引 → 索引中 → 可检索  
3. 索引完成后混合检索 / 问答应能命中内容  

### 9.2 索引重试

1. 模拟失败（如关闭 Embedding 8001 后上传）或使用失败记录  
2. 点击「重试索引」或调用 `POST .../reindex`  
3. 恢复服务后状态变为可检索  

### 9.3 对话与引用

1. 问答工作台提问  
2. 助手回复下方出现「参考来源」  
3. 点击来源文件名 → 打开预览  
4. 刷新页面后历史仍可从 MySQL 加载（管理员在「聊天记录」可查）  

### 9.4 管理端

1. 使用 `admin` 登录  
2. 打开「审计日志」→ 筛选 UPLOAD / CHAT 等  
3. 打开「运行监控」→ 刷新，应返回 200（不再 500）  

### 9.5 单元测试

```bash
mvn test -Dtest=ConversationServiceTest,DocumentAccessPolicyTest
```
